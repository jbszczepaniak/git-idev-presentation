First important thing

Which I think is useful

Add a following snippet to your ~/.bashrc file:


								parse_git_branch()
								{git branch 2> /dev/null | sed -e '/^[^*]/d' -e 's/* \(.*\)/ (\1)/'}
								export PS1="\u@\h \[\033[32m\]\w\[\033[33m\]\$(parse_git_branch)\[\033[00m\] $ "
And your terminal prompt will change from: 

						$
into: 

						(branchname)$
Good practise in local environment !
And must have on production server !

In bare repositories we don't have source code. They exist to put them on the server when you don't use any public repository.

When we try to push local changes to the remote repository, but git says that we can't because on the remote repository there are changes that we don't have yet, we have a problem.

What do we do then?

We do git pull which actually is equivalent to git fetch + git merge.

This means that git will take most recent commit of fetched changes, most recent commit of our work, last common commit of those two branches (changes fetched from remote repository are in fact another branch), and will perform three-way merge, which will result in creating merge commit.

The question is:

Do we really need commits for each such situation?

No

Why?

Because they make history of repository illegible.

What we can do?

Rebase!

To ilustrate this situation we will:

  1. create another local repository,
  2. clone remote repository,
  3. commit some changes in master branch and push them to remote repository,
  4. switch back to previous local repository,
  5. commit some changes,
  6. try to push them (they will be rejected),
  7. REBASE !

Everybody is happy because history is linear, but there is one golden rule about rebasing, which everyone should remember.

“No one shall rebase a shared branch” ~ Everyone about rebase

Why?

Remember, when we are rebasing we are changing history, which means that when someone who is working on the same branch as we, will try to push on branch we rebased and pushed, strange things will happen because git will think that history of the same branch is different, although in fact is the same.

But two scenarios we've covered are completely safe, namely:

  • git pull --rebase when you can't push to remote repository
  • git rebase on your private branch, to clear history

Fixing things

git stash

Useful for scenario:

  1. developer is working on given feature (on dedicated branch),
  2. manager comes and says: "dear developer, please change fonts on production branch",
  3. dev stashes changes,
  4. dev checkouts on another branch,
  5. dev changes font, commits a change, pushes it to the repo,
  6. dev then checkouts back to feature branch, gets his changes from stash,
  7. dev is working on given feature again.

Git stash works just like a stack, we can push changes on top of it, and pop them when necessary.

								
								~/presentation (new-feature) $ git status
								On branch new-feature
								Changes to be committed:
								  (use "git reset HEAD file..." to unstage)

									new file:   functions.c
								~/presentation (new-feature) $ git stash
								Saved working directory and index state WIP on new-feature: a07557e Contributors file
								HEAD is now at a07557e Contributors file
								~/presentation (new-feature) $ echo "#include < stdlib.h >" >> utils.c
								~/presentation (new-feature) $ git add utils.c
								~/presentation (new-feature) $ git stash
								~/presentation (new-feature) $ git stash list
								stash@{0}: WIP on new-feature: a07557e Contributors file
								stash@{1}: WIP on new-feature: a07557e Contributors file

We have on stash stack two elements, the latest at index 0.

Checking what changes are on which element of stack is done with:

                            		
									~/presentation (new-feature) $ git stash show -p stash@{1}
									diff --git a/functions.c b/functions.c
									new file mode 100644
									index 0000000..53c5fdf
									--- /dev/null
									+++ b/functions.c
									@@ -0,0 +1 @@
									+#include < stdio.h >

once again, git stash is a stack, with the most recent element at index 0.

                            	
							~/presentation (new-feature) $ git stash pop
							On branch new-feature
							Changes to be committed:
							  (use "git reset HEAD file..." to unstage)

								new file:   utils.c
							Dropped refs/stash@{0} (9abb1165605d667dd569651670a8dfffbdda8153)
							~/presentation (new-feature) $ git stash pop
							On branch new-feature
							Changes to be committed:
							  (use "git reset HEAD file..." to unstage)

								new file:   functions.c
								new file:   utils.c
							Dropped refs/stash@{0} (435763c5d1fbd8c8caf3cbeedf2f61184c7ceed3)

After two pops, stash stack is empty, and files came back to staging area (a.k.a. index).

We can commit those changes now.

							
							.~/presentation (new-feature) $ git commit -m "Add new important files"
							[new-feature fe99004] Add new important files
							 2 files changed, 2 insertions(+)
							 create mode 100644 functions.c
							 create mode 100644 utils.c

But wait.... We wanted to commit them in the master branch.

Not a problem at all.

Step one - checkout to desirable branch.

This git log command says "give me last commit from given branch" by the way.

It is ok, to give just the beginning of hash.

GIT RESET DEMISTIFICATION

What git reset is not about:

Git reset modes are:

  • --soft,
  • --mixed (default),
  • --hard,
  • --merge,
  • --keep.

We will briefly discuss first three methods.

Preliminary knowledge

Staging area is also known as: index, or cache

git reset --mixed

Resets HEAD to the specified commit and puts changes done after this specified commit in working directory.

How does the repo looks like?

							
							~/presentation (master) $ git lg | head -n 2
							* 2e173df - (7 hours ago) Add new important files - jedrzej (HEAD -> master)
							* 90792a8 - (3 days ago) Update README - jedrzej (origin/master)

							~/presentation (master) $ git status
							On branch master
							Your branch is ahead of 'origin/master' by 1 commit.
							  (use "git push" to publish your local commits)
							nothing to commit, working directory clean

git reset --mixed 

							
							~/presentation (master) $ git reset --mixed 90792a8
							~/presentation (master) $ git status
							On branch master
							Your branch is up-to-date with 'origin/master'.
							Untracked files:
							  (use "git add file..." to include in what will be committed)

								functions.c
								utils.c

							nothing added to commit but untracked files present (use "git add" to track)
							~/presentation (master) $ git lg | head -n 2

							* a07557e - (3 weeks ago) Contributors file - jedrzej (tag: v2, new-feature)

Files functions.c and utils.c are untracked now, because they were created after commit with hash: 90792a8.

git reset --mixed

git reset --mixed

git reset --hard

Resets HEAD to the specified commit and discards changes done after this specified commit

Repo looks like this:

							
							~/presentation (master) $ git lg | head -n 2
							* cf6018c - (3 minutes ago) Add new important files - jedrzej (HEAD -> master)

git reset --hard

Changes are gone. However in the case of mistake, we can recover from this thanks to git reflog. Git reflog is a history of things we've done to our repository locally, and it is pruned after 90 days (by default). that is why we can recover from mistakes most of the time.
